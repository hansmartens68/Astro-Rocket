Before you build anything on top of Astro Rocket, here's exactly what it ships with — no glossing over, no vague bullet points.


## The component library

Astro Rocket includes a full set of production-ready page sections. Every section is a standalone component you can drop into any page and configure via props.

**Landing page sections:**

- **Hero** — full-screen opener with headline, subheading, and call-to-action buttons. Supports a background image or video.
- **Features** — icon-led feature grid with configurable columns and layout styles.
- **FAQ** — two-column accordion. All items start closed; each opens independently.
- **CTA** — call-to-action band with headline, text, and button. Supports light and dark/inverted variants.
- **Testimonials** — quote cards in a configurable grid.
- **Pricing** — pricing table with tiers, feature lists, and a highlighted recommended plan.
- **Logo strip** — client logo row for social proof.
- **Newsletter** — email sign-up form with Resend integration.

**Layout components:**

- **Header** — sticky, transparent-on-scroll navbar with mobile drawer. Configured via `nav.config.ts`.
- **Footer** — four layout variants: `simple`, `columns`, `stacked`, and `minimal`.

All components are typed, documented, and built with Tailwind v4. They use CSS design tokens so colours and spacing stay consistent across the entire site.

## The blog

The blog runs on [Astro Content Collections](https://docs.astro.build/en/guides/content-collections/) with MDX support. Every post is a `.mdx` file in `src/content/blog/en/` with a typed frontmatter schema:

- **title** — max 100 characters
- **description** — max 200 characters, used as the meta description
- **publishedAt** — publication date
- **author** — displayed in the byline
- **image** and **imageAlt** — cover image with accessibility text
- **tags** — array of strings for filtering
- **featured** — mark a post as featured to surface it prominently
- **draft** — keep unpublished work out of production

The blog index shows all published posts with filtering by tag. Individual post pages include full Open Graph metadata, structured data, and a related-posts section. Cover images render with a brand colour overlay on both the blog grid cards and the individual post hero image. The overlay automatically uses the active theme's brand colour, so switching themes keeps everything visually consistent without any extra work. The overlay can be toggled off in `site.config.ts` via the `blogImageOverlay` flag.

## SEO layer

Astro Rocket includes a complete SEO implementation — not just meta tags.

**Structured data (JSON-LD):** Every page outputs schema.org markup. The home page includes `WebSite`, `Organization`, and `Person` schemas. Blog posts output `BlogPosting` with author, dates, and image. Structured data helps search engines understand what your site is and improves rich result eligibility.

**Open Graph and Twitter Cards:** Every page generates correct `og:` and `twitter:` meta tags. Blog posts use their cover image. Other pages use a configurable default OG image.

**Sitemap:** Auto-generated by `@astrojs/sitemap`. Every published page is included automatically.

**Canonical URLs:** Set correctly on every page to prevent duplicate content issues.

**Robots meta:** Individual pages can be marked `noindex` or `nofollow` via props.

**Verification tags:** Add your Google Search Console and Bing Webmaster verification codes in `site.config.ts` — no template editing needed.

## Dark mode

Dark mode is implemented cleanly, with no flash and no dependency on external packages.

The `<html>` element starts with `class="dark"`. An inline script in `<head>` — which runs before the browser paints anything — checks `sessionStorage` for a saved preference and applies the correct class immediately. No flash. No JavaScript framework required.

The theme toggle stores the user's choice in `sessionStorage`, so it persists during the session but resets on the next visit. CSS variables for all colours are defined once and remapped inside the `.dark` selector — switching the class repaints the entire site instantly.

## Contact form

The contact form connects to [Resend](https://resend.com) for email delivery. Set your `RESEND_API_KEY` and `CONTACT_FORM_ENDPOINT` environment variables, and form submissions arrive in your inbox.

The form includes client-side validation, accessible error states, a loading indicator, and a success confirmation. No third-party form services, no iframes, no external scripts loading on your pages.

## Navigation

Navigation is configured entirely in `src/config/nav.config.ts`. Add, remove, or reorder items without touching any layout files. The same config drives both the header and footer, so you define your navigation once and it stays in sync everywhere.

### Theme switcher

The header includes a built-in theme switcher — the light/dark/system toggle. On desktop it sits directly in the header navigation bar. On mobile it lives inside the drawer dropdown menu, keeping the header uncluttered at smaller screen sizes.

If you'd rather show a **CTA button** in that position — for example a "Start a Project" link — you can swap them out via the `Header` component's props:

```astro
<Header showCta cta={{ label: 'Start a Project', href: '/contact' }} />
```

Setting `showCta` replaces the theme switcher in the header with the CTA button. It's a one-line change — no layout editing required.

## Performance

Pages built with Astro Rocket consistently score 100 across all four [Google Lighthouse](https://developer.chrome.com/docs/lighthouse/) categories: Performance, Accessibility, Best Practices, and SEO.

This comes from shipping minimal JavaScript, pre-rendering all pages to static HTML, using optimised images through Astro's built-in `<Image>` component, and writing clean semantic markup throughout. Fast pages rank better, keep visitors engaged, and leave a better first impression.

## Deployment

Astro Rocket is configured for [Vercel](https://vercel.com) out of the box. Connect your repository, set your environment variables in the Vercel dashboard, and every push to `main` deploys automatically. All environment variables are documented in `.env.example` — none are required for basic functionality, so the site runs immediately after cloning.
